TL;DR. SONIC 是已训练好的全身平衡控制器：给它一个 64 维 FSQ 潜 token，它就吐出平衡、协调的全身动作（防摔/恢复全免费）。 要让 GR00T N1.7 按语言指令出舞蹈/武术，只需让 VLA 学会 prompt → token 映射，不做关节级行为克隆。 关键发现：UniversalTokenModule.decode() 在 Python 内独立可调 → 可在 Isaac 里闭环驱动，免 C++ / 免 DDS。 分三步走：Stage A（外部 token→WBC round-trip）→ Stage B（stub 把 prompt 路由到已知动作）→ Stage C（真 GR00T 微调）。 Skills live in the token space, not in GR00T. GR00T only needs to learn prompt→token; the WBC does the rest. decode() is callable in-process → close the loop in Isaac with no C++.

✅ 2026-06-03 更新：A→C 全链路已实跑通（见 §10）。GR00T N1.7 微调 8000 步，预测 token 对 GT 的 open-loop MSE = 0.0011（破「按动作均值预测」基线 ~35×）；7 条动作经 decode() 注入 SONIC，踢腿/舞蹈/macarena 与 GT 老师基本同步。但仅训练集内——新动作泛化被数据卡死，且注入是开环回放，闭环抗漂移仍未验。 UPDATE 2026-06-03: full A→C chain executed. 8k-step finetune, predicted-token open-loop MSE 0.0011 (~35× below the per-motion-mean baseline); injected tokens track the GT teacher. But training-set only — held-out is data-blocked and injection is open-loop replay; closed-loop drift untested.

1. 两条路线：数据侧蒸馏 vs 架构侧 WBC

把人形 VLA 做出来有两条独立路径，本文只讲架构侧（第二条）：

维度数据侧（蒸馏 / DART）架构侧（SONIC WBC）本文
思路teacher 策略 rollout → 录 (obs, 关节动作) → BC/DART 训 VLA 直接出关节VLA 只出 WBC 的潜 token；WBC 解码成平衡全身动作
VLA 学什么完整 29-DoF 关节控制 + 平衡（难，covariate shift）仅 64 维潜 token（轻）；平衡由 WBC 兜底
底座无，VLA 自己扛全部SONIC 预训练 WBC = 现成平衡底座
失败模式0.5s 摔倒（纯 clean BC 的协变量偏移）token 空间覆盖不到的动作（如高能武术）WBC 复现不了
对应记忆mimickit-to-vla-distill-plan本文 / gear-sonic-preview-setup

Two independent routes. Data side = clone joint actions (hard, covariate shift). Architecture side (this doc) = VLA emits the WBC latent only; balance comes free from the pretrained controller.

2. 当前状态：SONIC WBC 已跑通

已在本机 Isaac-eval 单进程跑通 SONIC 预览（G1 行走，GPU ~10G/30%）。当前执行的不是 VLA，而是 SONIC 释放的 WBC checkpoint：

  • 权重 sonic_release/last.pt，TRL/PPO actor-critic 训练
  • actor = UniversalTokenModule，backbone 内置 FSQ（finite scalar quantization）做 motion tokenizer
  • token 维度：num_tokens=2 × token_dim=32 = 64（token_flattened）
  • 解码器 g1_dyn：输入 ['token_flattened','proprioception'] → 输出 action（29-DoF 关节位置目标）
  • 当前回路：预存 reference motion（walk_forward_amateur_001）→ 跟踪。尚无 VLA。
环境侧已 6 块依赖补齐（easydict / loguru / open3d / vector_quantize_pytorch / trl==0.28.0）+ G1 全白修复（spawn 改 textured USD）。详见 gear-sonic-preview-setup 记忆与 patches/gear-sonic/。

3. 架构：潜 token 接力

4. 核心洞察：技能在 token 空间，不在 GR00T

SONIC eval 的流水线本质是：reference motion → encoder → FSQ token → decoder → 动作。

推论：给 eval 喂一段舞蹈/武术参考动作，WBC 现在立刻就能跳/打 —— 完全不用 GR00T。GR00T 的唯一工作是把前端的「参考动作」替换成「prompt + 视觉」→ token。运动技能本身已经在 WBC 的 token 空间里。
The motor skill is already in the WBC's token space. Replace the "reference motion" front-end with "prompt+vision". GR00T learns only prompt→token.

5. 技术命门：decode() 可调，免 C++

路径token → 动作怎么走依赖
官方 run_vla_inference.pytoken 经 ZMQ PUB（latent protocol v4）发给 C++ 控制环解码重 C++ deploy build + DDS + camera server
本方案（Isaac 内）UniversalTokenModule.decode("g1_dyn", {token_flattened, proprioception})["action"] 直接出 29-DoF轻 纯 Python，Isaac 进程内

decode()（universal_token_modules.py:692）是独立可调的；forward() 还支持 latent_residual_mode="pre_quantization_replace" —— 用外部潜替换编码器潜后再量化。两个注入口都能让外部 token 驱动 WBC。decode() is a standalone callable; forward() also accepts an external latent that replaces the encoder latent. Either is an injection point.

6. 落地计划：Stage A / B / C

为什么 stub 先行：真 GR00T 必须先注册 embodiment + 微调才能吐 SONIC token（基建活）。Stage A/B 用 stub 把管路与 WBC 可驱动性先验通（几小时），Stage C 再换真 GR00T。这正契合「先跑通，用 SONIC 当前会的动作」。

7. 关键接口：embodiment / 数据 / 微调

  • Embodiment：UNITREE_G1_SONIC，动作 78 维 = 64 motion token + 7 左手 + 7 右手。
  • Modality / features：repo 自带 gear_sonic/data/features_sonic_vla.py → get_modality_config_sonic_vla() 直接产出 meta/modality.json（observation.images.ego_view 640×… / observation.state 关节 / observation.eef_state 14 / action.motion_token 64 …）。不用自己造 modality.json。
  • 微调：Isaac-GR00T launch_finetune.py --base-model-path <N1.7> --dataset-path <...> --embodiment-tag UNITREE_G1_SONIC --max-steps 20k。N1.7 submodule 在 dependencies/Isaac-GR00T，env gr00t_n1d7，skill gr00t-4090-finetune 现成。
  • Isaac camera：eval 在 render_results=True 时挂 TiledCameraCfg（ego_view），Stage B/C 的视觉 obs 可直接取。

8. 拦路虎与风险

R1 · token 空间覆盖（最大风险）。SONIC 是平衡/locomotion 为主的控制器。高能武术（跳、踢、快速重心转移）可能超出它的训练分布 → 哪怕给对 token，WBC 也复现不了。Stage A 几块钱就能验：把一段目标动作 encode→decode round-trip，看 G1 跟不跟得住。
R2 · 动作来源。现仅 sample_data（1 条 walk）。"不同 prompt" 至少要几条语义清晰的动作：① SONIC 完整训练集（--training ~30GB SMPL，大概率含多样 locomotion）；② 把已有 LAFAN dance/fight retarget 成 robot_filtered 格式（工作量大）。
R3 · FSQ token 是离散的。GR00T N1.7 的 action 头是连续 flow-matching；SONIC token 是 FSQ 量化（每维有限档）。用连续头回归 64 维再量化，是否能学到正确 token？需验证（可能要把 token 当分类 / 学量化前的连续 latent）。
R4 · 观测/归一化对齐。WBC 训练时带 RunningMeanStd 归一化 + 特定 proprioception 布局。Stage A 的 in-process decode 必须喂同分布的 proprioception，否则动作漂移。

9. 留给评审的开放问题

  1. Stage A 的 in-process decode() 闭环，是否忠实于官方 C++ deploy 路径？有没有 obs 归一化 / 时序 / 控制频率的隐藏 mismatch？
  2. R3：FSQ 离散 token vs GR00T 连续 action 头 —— 正确的训练目标是 token 索引（分类）、量化后连续值、还是量化前 latent？
  3. 是否值得做 stub（Stage B），还是直接上 GR00T（Stage C）更省总时间？
  4. R1 验证：除了 round-trip，还有什么更强的「token 空间能否承载某动作」判据？
  5. 动作来源：先赌 SONIC 自带训练集含可用动作，还是直接投入 LAFAN→SONIC retarget 管线？
  6. 这条架构侧路线 vs 数据侧蒸馏（DART），在「出舞蹈/武术」这个目标上，谁的期望 ROI 更高？

10. derisk 复盘：全链路跑通 + MSE 0.0011（2026-06-03）

计划里的 Stage A→C 在本机一条龙跑通，架构侧路线 GO 坐实（限训练集内）。无 C++、无 DDS，全程 Python。The whole prompt→token→WBC chain runs locally end-to-end, no C++/DDS.

训练收敛：MSE 越训越低，不是天花板而是欠拟合

open-loop eval = 预测 token 对 GT token 的逐轨迹 MSE。两个朴素基线：按动作均值预测 ≈0.039、按全局均值预测 ≈0.048。

checkpointopen-loop MSE对比基线解读
2000 步（LR 已衰减完）0.065＞基线 0.039欠拟合（LR schedule 在 2000 就衰到 0，没学够）
8k run @ 20000.0167＜基线同步数、LR 不早衰 → 直接过基线
8k run @ 80000.0011破基线 ~35×继续降 → 证明是欠拟合不是天花板

MSE kept dropping to 0.0011 (35× below the per-motion-mean baseline) — the earlier 0.065 was an undertrained LR-schedule artifact, not a representational ceiling.

注入可视：GR00T 驱动 vs GT 老师

把 GR00T 预测的 token 经 decode() 注入 SONIC，Isaac 里 env_0 跑 GT token（老师）、env_1 跑 GR00T 预测 token（学生）并排对比。踢腿 / macarena / party-dance / lunge / squat / jump 等 7 条基本同步。

结论 / Verdict. 架构侧路线在「prompt→token→WBC 出平衡动作」上跑通且收敛。GR00T 确实能把动作语义压进 64 维 FSQ token，WBC 兜住平衡——验证了核心假设「技能在 token 空间」。
诚实边界 / Honest caveats（必须写清）：
  • 仅训练集内：只录了 7 条动作各 1 episode，eval/注入都在这 7 条上 → 本质是 7-ep 过拟合的「能记住」，不是泛化。
  • 新动作泛化 = 数据卡死：本地 demo_robot_filtered.pkl 只含这 7 条；更多动作要从 HF 下 ~30G robot_filtered。
  • 开环回放：注入的是预先 dump 的 token 序列，env 观测不参与 → 没测过「闭环里观测漂移后 GR00T 还稳不稳」，而这才是架构侧真正的风险点。
死路存档 / Dead-end： 想把 env_0 老师机器人涂绿做对比 → IsaacLab 机器人是 instanceable 资产，mesh 是只读 instance proxy（bind 0）；SetInstanceable(False) 能去实例化（bind 50 mesh）但破物理（"Failed to set DOF actuation forces"）。结论：用位置区分老师/学生，别涂色。

下一步：闭环活循环（唯一能用现有素材推进的真 derisk）

GR00T N1.7（transformers 4.57.3 venv）与 SONIC eval（Isaac conda）不能同进程，闭环只能走 server/client 桥：

  • 起 GR00T ZMQ server：复用 gr00t/eval/run_gr00t_server.py + server_client.py（wire 协议早已 debug，见记忆 gr00t-n17-leisaac-wire-debug），加载 checkpoint-8000。
  • 写 SONIC live-injector callback：复用现有 eval_callbacks 钩子，每步从活 env 组 obs（ego cam + proprio + projected_gravity + prompt）→ ZMQ 取 token → 替换 decode_input_dict["token_flattened"]。
  • 验：观测漂移下 GR00T 是否仍出可解码 token、WBC 是否仍兜住平衡。

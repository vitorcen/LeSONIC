LeSONIC · 架构侧 / Architecture route

动作串联：一句话时序驱动连贯动作

Action Sequencing — prompt-time-sequenced motion chaining over one live GR00T × SONIC session · 2026-06-09
TL;DR — 一个运行中的 Isaac 会话里,按时序切换 prompt,让 G1 连贯地做 蹲→走→踢→… 并循环,全程不重启 GUI / 不重启 GR00T server。机器人走到新位置就在那里做下一个动作。
One live session; the prompt is switched over time so the robot chains motions and loops, with no GUI/server restart. The robot walks to a new spot and performs the next action there.

关键认知 / Key insight:这是对一个记忆了 7 个动作的模型做 prompt 编排 (orchestration),不是复合指令理解;真正的难点不在"串联",而在调试过程暴露的 三个独立时钟(见 §4)。

1 · 它是什么 / What it is

底层架构沿用架构侧路线:GR00T N1.7 VLA 只吐 SONIC 的 64 维 FSQ motion_token,注入器 hook 住 actor_module.decode("g1_dyn", …) 用外部 token 覆盖,冻结的 SONIC WBC 把 token 解码成 G1 关节、负责平衡。动作串联在此之上加一层:把一串 prompt 段排成时间线,在同一个 live 会话里随时间切换。

能成立的底层支撑(实锤):GR00T server 无状态,prompt 每次 query 放在 observation 的 language 里发 → 运行中改 self.prompt,下一次推理就切,server / GUI 都不用重启。
The GR00T server is stateless — the prompt rides in each observation — so switching it mid-run needs no restart.

2 · 段内三窗口 + 演示流 / Per-segment windows & flows

注入器(vla_live_injector.py)按 timeline 推进,每段内部顺序:

窗口 window作用用于
settle段首注入中性/冻结上一姿态,平滑接缝所有非首段
bootstrap回放离线 dump token 把机器人带进一次性动作kick/walk/jump(从站姿启不动的)
live跟 GR00T server 实时吐的 token自持动作(squat/lunge/dance/macarena)

full_bootstrap:walk 段整段回放走路 token(连续行走当"连接器"),因为 7 个 demo 里没有独立 run,locomotion 只有 walk。

gear_sonic_live_demo.sh 自动起 GR00T server(幂等,缺则加载约 1 分钟)再开 viewer——因为 server / viewer 在不同 env,必须编排:

4 · 调试实录:三个独立时钟 / The three independent clocks

"动作串联"代码本身不难;真正的硬骨头是调试过程暴露的三个独立时钟。GUI 上反复出现"机器人闪回原点"和"卡死",根因都在时钟。这一节是本文最有价值的部分。
The chaining code was easy; the hard part was three independent clocks surfaced while debugging the "flash back to origin" + "freeze" symptoms.

1-frame 硬传送回参考原点 (0,0,0.79) → 绕过 dones!我的 dones 探针完全看不到 → 误判"没有 reset" walk 把机器人推到 −1.9m;8.5s 一到就被传送回 0 → "闪回原点作下一个动作"

4.1 诊断顺序(以及我错在哪)

  1. 症状:flow2 里机器人"闪回原点作接下来的动作",且失去"走到别处作动作"的能力。
  2. 错误假设:我加 dones 触发日志 → RESET=0,注入器 global_t 单调 → 我判"不是 reset",转而猜"WBC 锚点拽回"。
  3. 探针 root XY:机器人稳定在 (−20,−76)(随机 terrain 的 env-0 子原点),离世界原点 80m。我先以为是"相机闪回世界原点"。改 terrain_type=plane 把机器人放回 (0,0) —— 没解决(只是让闪点和机器人重合一会)。
  4. 每 50 步记 root(覆盖 walk 段):walk 把 root 推到 −1.87m;下一段 macarena 一开始就弹回 ~0。时序铁证:squat clip=424 帧 ≈ 弹回发生的 step。
  5. 真凶:读 _update_command(docstring 明写 "handle episode wrap-around")——clip 播完调 _resample_command → write_root_state_to_sim 硬传送回原点,不设 dones。
三方独立验证 / Triangulated. 让 codex(gpt-5.5)+ mimo-v2.5-pro 两个模型独立批判,都独立指向同一处并否掉我的"WBC 拽回":
  • "'No dones' does not rule out a root write … `_resample_command` writes root via `write_root_state_to_sim`. That is a soft command reset, not a Gym termination. Your dones probe missed the main suspect."
  • "g1_dyn decoder inputs are only token_flattened + proprioception — it does NOT include motion_anchor_pos_b. Don't sell this as an XY anchor spring."(= 我的"锚点拽回"框架错了)
  • 最大盲点:"two clocks — you disabled the Gym episode clock and left the TrackingCommand motion clock armed."

4.2 修复:freeze 而非 loop

修法 = 当机器人由外部注入 token 驱动时(参考只是惰性脚手架),冻结参考时钟:不推进、不重采样,边界永不触发。SONIC_NO_REF_RESAMPLE=1 环境变量门控(patches/gear-sonic/0006):

# gear_sonic/envs/manager_env/mdp/commands.py · _update_command()
if os.environ.get("SONIC_NO_REF_RESAMPLE", "0") != "0":
    pass                       # freeze: 不推进 time_steps、不调 _resample_command
else:
    self.time_steps += 1
    env_ids = torch.where(self.time_steps + self.motion_start_time_steps
                          >= self.motion_lib.get_time_step_total(self.motion_ids))[0]
    self._resample_command(env_ids)   # 原行为:clip 末重采样(会传送机器人)
踩过的二次坑 / The fix that broke it. 我第一版用"循环时钟"(time_steps = −motion_start_time_steps)而非冻结。RSI 起始帧非零时 time_steps 变负 → 参考帧索引越界 → NaN → 物理炸 → GUI 卡死(用户报"原本闪回的时间点现在卡死")。
正解是冻结(codex/mimo 都推荐 freeze/clamp 而非 loop):停在 init 的有效帧,无负索引、无 buffer 不一致。

7 · 诚实边界 / Honest scope

  • 是编排,不是理解。一句话"蹲走踢"是状态机按时序换 prompt串起来的,模型并没解析这句复合指令。真·组合 prompt 需要先补 P1 obs history/phase clock + 过 held-out 泛化闸门(见 critique roadmap)。
  • 参考是惰性脚手架。本路线下机器人完全由注入 token 驱动,加载的 reference 只为 env 初始化;冻结它的时钟不影响动作质量。
  • 实时 viewer 有推理 stall。平滑演示视频应走 render-to-mp4,不要录抖动的实时画面。
  • 仍 RELAX(终止阈值放宽)。这是动作可玩性的 demo,非 fall-resistance 验证。

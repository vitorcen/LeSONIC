给 SONIC 弄舞蹈/武术动作：三条路哪条靠谱？

30G SMPL 重定向 vs LAFAN 重定向 vs 本地 deploy 参考动作 —— 实测取证后的结论
Where do dance / martial-arts motions for the SONIC WBC come from? Evidence-based comparison. 2026-06-03.
TL;DR. 调研推翻了"下 30G SMPL"和"retarget LAFAN"两个先验。第三条路胜出： 仓库本地 gear_sonic_deploy/reference/example/ 已有 SONIC 自己的 demo 动作 —— macarena、dance_in_da_party（舞）、neutral_kick（踢=武术）、lunge、jumping、squat、walk-360， 是 G1 29-DoF 真实轨迹（CSV）。而 gear_sonic/data_process/convert_soma_csv_to_motion_lib.py 正好把这种 CSV 一条命令转成 Isaac-eval 要的 robot_filtered pkl；外加 smpl_motion_file 可传 dummy（源码已支持），连 SMPL 都不用配。 两个独立评审（codex gpt-5.5 + mimo）也都说"先用 SONIC 现成动作，别先投 retarget"。结论一致：走路 A。
Local SONIC demo motions (dance/kick/lunge) already ship as G1 CSV; a ready converter turns them into the eval's pkl format, SMPL optional. Don't download 30G, don't retarget LAFAN — yet.

2. 路 A 详解：本地已有，一条命令

动作清单（gear_sonic_deploy/reference/example/，每条一个目录，内含 joint_pos.csv / body_pos.csv / body_quat.csv / joint_vel.csv / body_{lin,ang}_vel.csv + info.txt）：

  • macarena_001__A545 · dance_in_da_party_001__A464(+_M) —— 舞蹈
  • neutral_kick_R_001__A543 —— 踢（武术向），joint_pos 形状 (165, 29)
  • tired_forward_lunge_R_001 · tired_one_leg_jumping_R_001 · squat_001 · walking_quip_360_R_002

转换器 gear_sonic/data_process/convert_soma_csv_to_motion_lib.py 的 docstring 明说：输入「含 joint_pos.csv/body_pos.csv/body_quat.csv 的目录」或「deploy PKL」，输出 motion_lib 训练格式 (root_trans_offset, pose_aa, dof, root_rot, fps) —— 与本机 robot_filtered 实测字段逐项吻合。

# 1) deploy CSV → robot_filtered pkl
#    ⚠️ 转换器默认 --fps 30，但 SONIC eval 的 target_fps=50 → 必须显式 --fps 50。
#    且要先确认 deploy CSV 源采集帧率：源是 30fps 而当 50 用 → 动作快 1.67x、踢腿变形。
#    源非 50 时用 --fps_source <真实fps> 降/升采样。
python gear_sonic/data_process/convert_soma_csv_to_motion_lib.py \
    --input gear_sonic_deploy/reference/example/neutral_kick_R_001__A543 \
    --output data/kick_robot.pkl --fps 50

# 2) 跑 Isaac-eval（SMPL 传 dummy，免配套；转换器已输出 smpl_joints=zeros 占位）
python gear_sonic/eval_agent_trl.py +checkpoint=sonic_release/last.pt +headless=False ++num_envs=1 \
    "++manager_env.commands.motion.motion_lib_cfg.motion_file=data/kick_robot.pkl" \
    "++manager_env.commands.motion.motion_lib_cfg.smpl_motion_file=dummy"
评审修订（codex gpt-5.5 + mimo，二者均实测核实）：
  • smpl_joints 不是坑（误报）。转换器 :301 已输出 smpl_joints=zeros(T,24,3) 占位；mimo 追源码证明 smpl=dummy 时 g1 encoder 只吃 command_multi_future_* + motion_anchor_ori（robot FK 派生），根本不读 smpl_joints。安全。
  • 真正的坑是 fps。--fps 默认 30，必须传 50（eval target_fps=50），且要确认 deploy CSV 源帧率，否则动作变速变形。这是路 A 唯一实际拦路点。
  • 动作是 13 条不是 7 条（含 _M 变体：macarena×2 / dance×2 / kick×2 / lunge×2 / jump×2 / squat / walk）。
  • 降调结论。deploy demo 可能是精选"能跑"的动作 → 路 A 全过只证明 WBC 吃这些动作，不等于能泛化任意武术。是 go/no-go，不是"覆盖武术需求"。
  • 新增路 D（mimo，先做）：先用已有 walk robot_filtered pkl 跑通 eval 验证管线（零转换）——本机预览已跑过 walk，管线实际已验——再换 kick/dance，把"管线 bug"与"动作 bug"分开排查。
  • 扩容走路 E（两者一致）：路 A 之后不下 30G SMPL，而是取 Bones-SEED 的 G1 retargeted CSV 子集（按 dance/boxing/kick/jump 关键词筛）走同一转换器；比路 B 省、比路 C 无物理 gap。优先级：A → Bones-SEED G1 CSV 子集 → LAFAN/C → 全量 30G SMPL。
为什么最靠谱：动作是 SONIC 自己的 deploy demo（官方随附、格式同源、有官方转换器、SMPL 可 dummy）→ 最适合第一轮验证。注意（评审降调）：deploy demo 可能是精选"能跑"的动作，路 A 全过只证明 WBC 吃这些动作，不等于泛化任意武术 —— 是 go/no-go 信号，非"覆盖武术需求"。SONIC's own demo motions: ships with the repo, same format, official converter, SMPL optional — best for round-1 validation. But demos may be cherry-picked: passing Path A is a go/no-go, not proof of arbitrary martial-arts coverage.

3. 路 B：30G SMPL / Bones-SEED

HF nvidia/GEAR-SONIC 的 bones_seed_smpl/（~30G）是 SMPL 种子数据，解压到 data/smpl_filtered。注意转换器示例里也有 --input /path/to/bones_SEED/g1/csv/210531 —— 说明 Bones-SEED 要先重定向成 G1 CSV才能进转换器。即：下 30G SMPL → SMPL 重定向到 G1 CSV → convert_soma → robot_filtered，比路 A 多一层重定向管线，且「种子」是否含舞蹈要抽样确认。

何时才走 B：Stage A/B 用路 A 的 7 条 demo 验通架构后，需要大规模多样动作训 GR00T（Stage C）时，再考虑 Bones-SEED 全量。现在投是过早优化。

4. 路 C：LAFAN → robot_filtered

项目已有 LAFAN dance1_s2 + fight（MimicKit 训练用，已重定向到 G1）。优点是动作明确、可控；缺点：无现成转换器（要写 MimicKit motion → robot_filtered 的字段映射），且 LAFAN 是动画数据非物理，重定向到 SONIC 后可能足滑/失衡 → 需二次物理优化。两个评审都指出这是「动画→物理」gap，不应在架构未验证前投。

6. 两模型评审的收敛意见（上一轮，架构文档）

  • codex gpt-5.5：「先赌现成 SONIC motion corpus / gear_sonic_deploy/reference/example；现成 macarena/kick/lunge/jump 足够做 Stage A/B 的 go/pivot。LAFAN→SONIC retarget 应在 token 空间承载性验证后再投。」—— 本调研正是顺这条线索查实的。
  • mimo：「先赌 SONIC 训练集（格式原生兼容，不需 retarget）。LAFAN 是动画非物理，retarget 后要再过物理优化。但若 SONIC 集不含武术/舞蹈，就必须上 LAFAN。」
  • 二者一致：先用 SONIC 现成动作；retarget 是最后手段。本调研把"现成动作"具体定位到了 deploy reference + 现成转换器。

7. 推荐与下一步

走路 A。第一步把 neutral_kick + 一条 dance 用 convert_soma_csv_to_motion_lib.py 转成 robot_filtered，跑 Isaac-eval（smpl=dummy）—— 这同时是架构侧 Stage A / Milestone 0 的去风险实验：看 WBC 能不能跟住踢/舞而不摔。跟得住 → token 空间承载得了 → 架构侧绿灯。

8. 留给评审的开放问题

  1. 路 A 的 convert_soma_csv_to_motion_lib.py 输出缺 smpl_joints 字段（它由 SMPL 派生），eval 在 smpl=dummy 下是否真的不需要它？会不会 g1 encoder 仍隐式依赖 smpl_joints？
  2. deploy 参考动作的 fps / 关节顺序（MJCF actuator order）与 robot_filtered 约定是否完全一致？转换器是否已处理重排？
  3. 只有 ~7 条 demo 动作，够不够 Stage B 的 prompt 多样性？还是 Stage B 就得上路 B 的 Bones-SEED？
  4. 路 B 的「SMPL→G1 重定向」具体用 repo 哪个工具（motionbricks？process_dataset.py？），代价多大？值不值得为 Stage C 提前铺？
  5. 若路 A 验证 WBC 跟不住高能武术（踢/跳），是该退回数据侧蒸馏，还是先在 SONIC token 空间做 reachability 分析？

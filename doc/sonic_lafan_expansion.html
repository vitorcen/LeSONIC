LAFAN × SONIC-VLA 动作扩容能成吗？
Can LAFAN motions expand the SONIC VLA? — a three-model brainstorm

LeSONIC · 2026-06-09 · 头脑风暴：Claude Opus 4.8 + Codex GPT-5.5 + MiMo-v2.5-pro（独立评审）· 本文经 Codex 复审

TL;DR（一句话结论 / one-line verdict）
「融合（fusion）」是错词。MimicKit-G1-LAFAN 是一组 29-DoF DeepMimic PPO 策略（直接吐关节角）， SONIC-VLA 是 GR00T → 64 维 FSQ token → 冻结 WBC 解码——两者动作空间/架构都不同，权重无法合并。 LAFAN 唯一的入口是 SONIC 的 token 空间，而它被一道硬闸门卡住：冻结的 WBC 能不能在物理上跟住这个动作。 GR00T 技术上能吐任意连续 64 维值（decoder 照收），但只有落在 decoder 训练过的 FSQ 流形附近的 token 才会解出稳定有用的动作 （这正是部署侧要 snap-to-grid 的原因）——所以训 GR00T 去拟合「WBC 跟不住的动作」的 token，得到的也只是不稳的行为。
"Fusion" is the wrong frame. The MimicKit policy (joint-space PPO) cannot be weight-merged into the SONIC VLA (FSQ-token + frozen WBC). LAFAN can only enter via SONIC's token space, gated by whether the frozen WBC can physically track it. GR00T can emit any continuous token, but only tokens near the decoder's trained FSQ manifold produce stable behavior.

➡️ 2026-06-09 实测更新（见 §6.5）：严格 WBC 跟踪筛查真跑了——LAFAN fight（含高踢腿）全过 ✅（progress 1.0、 全局漂移 372mm 与 in-dist 同档），jumps 物理上做不出 ❌（放宽终止也偏 2.3m），run/外部 dance 边缘。 **规律：SONIC WBC 是接地平衡控制器——接地动态动作跟得住，弹道腾空跟不住。** 这部分推翻了「LAFAN 整体跟不住」， 变成动作类型级的 go/no-go：fight 是已验证可加的 VLA 新动作。当天就走通了完整管线——录 token→8 动作数据集→finetune→ fight open-loop token MSE 0.0021，落在原 7 动作区间正中（比基线低 18.7×）= fight 已真正进 VLA（详见 §6.5③）。 唯一悬而未决的仍是闸门② held-out 泛化（对所有动作，是记忆 vs 泛化的真问题）。

1 · 为什么「融合策略」是伪命题
Why "fuse the policies" is a non-question

两个产物的数据结构根本不在一个空间里 —— 这是问题的本质（"好程序员关心数据结构"）：

可复用的只有两样东西，都跟 MimicKit 的策略权重无关：

  • LAFAN 动作数据本身（公开，retargeted G1 参考已在 HF cache）—— 根本不需要任何策略。
  • （可选）MimicKit 策略当数据生成器：让训好的策略在 sim 里 rollout，得到「物理上可达」的 G1 轨迹，再喂 SONIC。 — policy only as a feasibility oracle, never as weights.

2 · 四条可能的路 + 排名
Four possible routes, ranked

路 / Route做法是否真扩 VLA成本排名

Codex 的关键纠偏：把 Route A 叫对名字。
A 的第一步不是「扩 VLA」，而是「WBC 能力筛查」。 "This is route A, but call it what it is: WBC capability screening, not VLA expansion. If strict tracking fails, GR00T training is irrelevant."
冻结 WBC 是天花板：WBC 跟不住的动作，其 token 区域 decoder 解不稳，GR00T 学了也只是不稳行为。所以先验证 WBC，再谈训 GR00T。

3 · 扩容优先级阶梯（LAFAN 为什么排最后）
Expansion ladder — why LAFAN is last

LAFAN 排最后有三条具体理由（MiMo 给的；其中①②在 LeSONIC repo 内核实，③源自 MimicKit 训练记录、非本仓可验）：

  1. WBC 分布 gap 是真的。 WBC 训练集是 motion_lib_bones_seed/robot_filtered（dance/loco 为主）。 LAFAN 的高踢腿/跳跃是动画数据，未经物理校验，WBC decoder 从没见过这些区域的 token → 即便 g1 encoder 吐得出 token， g1_dyn 解出来可能是垃圾。
  2. 没有现成 converter。 convert_soma_csv_to_motion_lib.py 支持 SOMA CSV/PKL、deploy PKL、 以及 Bones-SEED flat CSV（case #4/#5）——所以②不用写新 converter。但 LAFAN 是 MimicKit motion schema， 要从头写 MimicKit→robot_filtered 转换（字段映射 + fps 校正）。
  3. MimicKit 自己的 run 只到 63% 跟踪（外部 MimicKit 训练记录 mimickit-lafan-fight-training-plan，本仓无法直接核验）。 若它自家 PPO 都跟不满 run，说明参考轨迹本身部分不可行——把不可行参考喂 SONIC，只会产出不可行 token。
反过来看「②为什么便宜」：converter 已经原生支持 Bones-SEED flat CSV （load_bones_csv / BONES_CSV_JOINT_NAMES），和 WBC 训练分布同源 → 分布/转换风险最低、零新代码。 （注意：同源≠零风险，每条新 clip 仍可能严格跟踪失败，要逐条过筛查。） 按 dance/boxing/kick/jump 关键词筛 50-100 条，就能在碰 LAFAN 之前把动作词表做大。 BONES-SEED reuses the existing converter and the WBC's own training distribution — lowest distribution/converter risk and no new code (not zero risk: each clip can still fail the strict screen).

4 · 两道必过的闸门（在碰 LAFAN 之前）
Two gates to pass before LAFAN

闸门 1 · 严格 WBC 跟踪筛查（RELAX 必须关）

Codex 盲点：别用 demo 设置验证。 我们的动作串联脚本 （gear_sonic_sequence.sh）为了 demo 连贯性显式放宽了形变/摔倒阈值、关掉 time_out—— 这对 go/no-go 筛查是毒药。RELAX/sequencing settings are poison for a feasibility screen. 筛查必须：RELAX=0、原生终止阈值、headless、无相机、不录 token、不起 GR00T。

判据来源已存在（Codex 指出）：im_eval_callback.py 报 terminated / success_rate / 逐动作 progress（顶层聚合为 success/progress 率）/ mpjpe_g（全局 MPJPE，mm）。 严格终止用 release 自带的 base_adaptive_strict_ori_foot_xyz（不要用放宽的 tracking/eval.yaml 或脚本里的 RELAX=1→阈值 99）。 MiMo 盲点：必须先定阈值再跑——建议 terminated=No 且 progress 接近 1.0 且 mpjpe_g 低于 in-distribution 基线的某个倍数。

闸门 2 · VLA held-out 泛化（LOMO）

当前 VLA = 7 动作 × 1 ep，纯记忆，无 held-out。MiMo 盲点（也是 Codex 共识）： 在一个不能泛化的 VLA 上加 LAFAN token，等于「把查找表做大」——更多 token 进、同样的记忆出。 P0 #2 的 make_holdout_split.py（leave-one-motion-out）必须先过：held-out MSE 若爆炸， 说明 VLA 学的是 label 不是 skill，扩词表没意义。

MiMo 的灵魂拷问：你到底要「扩 VLA」还是「更多机器人行为」？
"Do you need to expand the VLA, or do you need more robot behaviors?"
• 要「demo 里多几个动作」→ Route C 双控制器最便宜最诚实（但 VLA 仍只会 7 个）。
• 要「一个 VLA 会很多事」→ 先解决泛化（每动作多 ep + held-out），再扩词表。

5 · 被抓出的盲点（点名感谢两位评审）
Blind spots caught

盲点谁抓的纠正
离线 encode() 是「免费捷径」Codex + MiMo 不是。encoder 要的是 command_multi_future_nonflat + motion_anchor_ori_b_mf_nonflat 这类精确的 robot-FK 参考特征，不是 raw LAFAN/SMPL。仍要构造精确 FK 特征，且 token 只有 WBC 解得出才有意义 → 还是要闭环验证。
FSQ levels = 16Codex + MiMo 错。配置是 num_fsq_levels: 32（all_mlp_v1.yaml:34）；部署侧 snap grid 是 17 个值 k/16, k∈[-8,8] = {-0.5…0.5}（0004 patch 实证）。code 是 FSQ-32，部署 token 值落 1/16 网格——两层别混。
RELAX 设置可用于 go/no-goCodex毒药，见闸门 1。
「更多动作」≈「更好 VLA」MiMo不等价。VLA 的问题是泛化（每动作 1 ep），不是词表大小。
MimicKit 63% run 跟踪无所谓MiMo红旗：参考本身部分不可行 → 不可行 token。
动作串联 SETTLE=40（0.8s）seam 够用MiMo 对高能量 LAFAN fight 可能不够：上一段踢腿还在减速，下一段就启动了。高能量动作要加长 settle。

6 · 第一个实验（一条 clip 转好后约 30 分钟，定生死）
The single first experiment

两位评审给的实验几乎一样，合并成最稳的版本：先用 in-distribution 的 kick 验证「严格筛查 harness + 阈值」本身， 再打 LAFAN fight 这个真风险。无 GR00T、无 recorder、无 MimicKit、无 RELAX、无相机。 注意：B 步要先有一条转好的 LAFAN robot_filtered（写 MimicKit→robot_filtered converter 或手转一条最短 clip）才能跑—— 「30 分钟」指的是 clip 转好之后的那次 eval。

🎯 实测推翻「一刀切 LAFAN 跟不住」，给出一条物理规律：
SONIC WBC 是接地的平衡/接触控制器 → 接地的动态动作（fight 高踢腿/lunge）跟得住， 弹道/腾空动作（jumps 双脚离地、身体走抛物线）跟不住，带飞行相的 locomotion（run）部分跟、漂移大。 SONIC WBC is a grounded balance controller: it tracks dynamic grounded motion (fight kicks) but not ballistic flight (jumps). This refutes the blanket "LAFAN won't track" — fight is a validated GO.

对原问题的直接回答：「能不能用 LAFAN 给模型加更多动作」——fight 可以（WBC 严格筛查全过、token label 有效）， 走的就是和现有 7 demo 一模一样的管线（录 token → finetune GR00T）；jumps 不行（WBC 物理上做不出腾空）； run / 外部 dance 边缘（要么调终止、要么 Route D 重训 WBC）。这是动作类型级的 go/no-go，不是「LAFAN 整体不行」。

8 · 技能化拆分 + flow3（2026-06-09 晚，含三模型二次评审）
Segmenting into promptable skills for flow3

目标：把 fight/run 从「一坨」拆成一个个连贯招式（按速度谷自然边界，非固定时间），每个当可单独 prompt 的技能， 串成 flow3（跑一段→作一个技能→再跑→下一个技能，循环）。方法论：拆段 → 严格 WBC 筛查（客观过滤）→ 录 token+训 → GUI 逐个演示留好的 → flow3。

8 个细技能（jab/cross/hook/uppercut）❌ 太细 → 砍到 3-4 粗类（punch/kick/defensive/footwork）。ego 只看地面，细拳从画面无法区分，prompt 唯一信号 + 轨迹近同 → mode collapse 几乎必然。
flow3 串接 = 学会组合？❌ 是编排（orchestration）非学习的组合。真瓶颈是转场：需指定 guard 为中性枢纽，所有技能回它（否则 8×8 转场吸收不了）。
单 clip「5 样本」同一表演者同一 take 的相关重复 = 零泛化多样性，只降方差。
held-out 泛化仍是没过的真闸门；本轮按用户决定推迟到 flow3 之后当大任务（当前明确是 demo 性质=记忆编排，和已发布 demo 同质）。
工具债gr00t_dump_pred_tokens.py 硬编 range(7)+旧 key；converter 一段一 prompt 无转场上下文 → 扩段前要改。

来源通过关键发现
fight11 / 14大多接地招式跟得住（mpjpe_l 13-41mm）；3 段栽（seg030/040/062，progress 0.32-0.88）。
run14 / 14 全过 ✅🎯 推翻「run 不值得拆」：整段 15s run 严格垮@61%，但短段(2-4s)全过——短时长累积漂移不够触发终止。低位移转身段(0.01-0.27m travel)跟得最干净(mpjpe_g 43-60mm)；高位移冲刺段(8m)也不摔但全局漂 ~800mm(跟住步态、落后地面覆盖)。
结论修正：run 短段可用（尤其转身/中速）。 两评审都判「run 不值得拆、复用 walk」，但段级实测 14/14 过—— 整段失败 ≠ 段级失败，关键是时长累积。flow3 的 locomotion 因此可以用 LAFAN run 的转身/慢跑段（比单一 walk 更丰富）， 高速冲刺段虽过但漂移大、demo 质量待 GUI 验。Segment-level run tracks 14/14 even though the full clip failed — short duration doesn't accumulate the drift; turns/jogs are cleanest. Refutes "don't segment run."

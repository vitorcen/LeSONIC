SONIC-VLA 架构治本：观测加速度 + 历史窗 重训方案

Architectural cure for the cold-stand freeze: add proprioceptive velocity & observation history to a single-frame token VLA.

TL;DR
问题：GR00T-N1.7 SONIC 是单帧、无速度的 token VLA。冷/漂移站立时 obs 退化 → prompt 条件塌缩 → kick/jump/walk 冻结（conditioning collapses at OOD static stand）。Tier-1 重训修好了 in-dist 站立，Tier-2 CFG 实测救不了OOD（放大的是退化噪声方向）。

方案：给 observation 加 ① 关节速度（数据重建时差分）② 历史窗 T=2~3（过去帧）。
关键发现：MODERATE GR00T-N1.7 原生支持历史窗与扩展 state，无需深度架构改动 —— state encoder 输入维 = max_state_dim × state_history_length，前向已按 [B,T,D]→[B,1,T·D] 处理，无硬编码 squeeze。max_state_dim=132 是 padding 维，当前 SONIC state 仅 46 维，加速度后 ~89 维仍有余量。

诚实边界（经 codex gpt-5.5 评审修正）：速度在纯静止站立=0，且3 帧 OOD 历史仍是 OOD（不会"摊平"持续漂移姿）→ velocity/history 无法修复冷启动那一次失败的首预测，只能改善启动后的承接/sustain 与一般闭环质量。walk（首 chunk 已 tok_std=0.171=hold）大概率救不了。3 OOD frames stay OOD; velocity=0 at static stand → cannot repair the failed FIRST prediction, only sustain after motion begins.
★ codex 评审颠覆性结论：架构改不是本 onset 冻结的最高 ROI。 正确投资顺序 = ① 切换前 re-anchor 到已演示站立姿（最高可靠/最低训练风险）→ ② Tier-3 onset priming（冷切换的针对性疗法）→ ③ velocity+history T=2 重训（提升一般闭环质量与 sustain，非 onset 首选）。 本方案保留为③，但建议先做①②。

3. 方案设计 Design

3a. 关节速度 Joint velocity

  • 数据侧：sonic_vla_resample_build.py 逐 episode 算 vel[t]=(j[t]−j[t-1])·fps（首帧补 0），拼到 observation.state 角度之后。
  • 新增 state key（7 关节组的速度，43 维；projected_gravity 是常零不加速度）：left_leg_vel … right_hand_vel。
  • embodiment unitree_g1_sonic state.modality_keys 追加这 7 个 vel key（embodiment_configs.py:72-84）。
  • 重算 dataset statistics.json（含新 key 的 min/max/mean/std）；processor 自动归一化拼接。

3b. 历史窗 Observation history

  • state delta_indices: [0] → [-2,-1,0]（T=3）或 [-1,0]（T=2）。
  • model config state_history_length: 1 → 3（configs/model/gr00t_n1d7.py:80）。state encoder 输入维自动变 132×3，从零重训该 MLP（finetune 本就重训 head）。
  • video 保持 T=1：ego-view 帧间几乎不变，加 video 历史 = 多倍 VLM token，成本高收益低。先只做 state 历史。

3c. 部署侧（injector）Deploy-side

  • vla_live_injector.py 需缓存最近 K 帧 state，发 [K,D]；并在线算速度 (q[t]−q[t-1])·hz。
  • 首 K-1 帧用复制首帧补齐（warmup）。
  • 训练/部署的速度定义（fps×差分）必须一致，否则归一化错位。

4. 机制诚实分析 Does it really fix onset? (honest)

速度在纯静止站立 = 0；且 K 帧重复 OOD 姿仍是 OOD（codex 反驳了原稿"历史摊平 OOD"的说法 —— 三帧 OOD 不会变 in-dist）。所以在精确零速度冷启动那一刻，state（角度+速度+历史）几乎无法区分 "hold" 与 "即将 kick" —— onset 决策仍只靠 prompt。velocity/history 无法修复冷启动的首次失败预测。

它真正买到的（已收窄为两件，且都在动作起来之后）：

  1. 承接/sustain：一旦动作起了一点点（靠 prompt 或 priming 触发），速度≠0 + 历史轨迹 → 模型读到"正在运动"→ 续出 onset token，而非塌回 hold。"自锁 hold 吸引子"仅在首预测失败之后成立（hold 输出保持零速度→后续 obs 留在同一盆）—— velocity/history 防的是这个二阶衰减，不是一阶启动。
  2. 一般闭环质量 / 相位估计 / 瞬态噪声恢复：多帧上下文让 token 序列更平滑、对单帧传感噪声更鲁棒。
对 walk：大概率无效。 walk 首 chunk 已是 hold（tok_std 0.171），预测器尚未吐出任何运动信号 → 无运动可承接。velocity/history 能防 walk 起来后衰减，但不能让 walk 起步。walk 的起步缺口要靠数据侧 onset 教学或 priming。

结论：velocity/history 是闭环质量改进，不是 onset 冻结的解药。onset 必须靠 ①re-anchor 或 ②priming 或 数据侧 onset 前缀。三者与本架构改互补，但优先级在前。

6. Silent blockers（codex 评审挖出，实施前必处理）Implementation blockers

这些是会静默破坏训练/部署、不报错的坑，比"风险"更硬。
#blocker处理
B1负历史 = 时间泄漏：allow_padding=False 且 dataset 采样 step 0 时，iloc[-2/-1] 会从 episode 末尾取帧（sharded_single_step_dataset.py:40）→ 严重时序泄漏显式排除前 K-1 索引 / 开启并测试 padding；必须验证首帧取到的是同 episode 的复制而非别处
B2warm-start 非自动：改 history → state_encoder 权重 shape 变 → checkpoint 加载拒绝不匹配 key 而非选择性重初始化（setup.py:107）写自定义 ckpt 迁移：只继承 VLM+action head，state_encoder 从零
B3config 覆盖可能无效：改源码默认 state_history_length 未必盖过 checkpoint 里存的值验证实例化后的 config，不是源码默认
B4表征断层：随机 state_encoder 喂预训练 action head = 大表征不连续；"几千步"无依据分阶段 LR / 先冻 downstream 数百步再放开；预算上调，别按 6ep 想当然
B5速度语义：不是简单 ×FPS。需精确关节序、基于 timestamp 的 Δt、episode 边界 reset、滤波/clip、训练与部署同采样/控制率语义用 timestamp 算 Δt；首帧 vel=0；部署在线速度与训练逐字段对齐（最易 silent bug）

7. Gravity 审计（codex 挑战 → 已证 doc 原稿过时）Gravity audit

codex 质疑"proj_grav 恒零 bug"过时 —— 实测确认 codex 对。
  • resampled3 训练数据 observation.projected_gravity = 真值（episode0 首帧 [0.076, 0.025, −0.997]，min/max −1.0/0.076，非零）。builder sonic_vla_resample_build.py:118 已 compute_projected_gravity(q[t])。
  • 部署侧 VLA_ZERO_GRAVITY 在任何 launcher 都没设 → injector 发真值（vla_live_injector.py:399），与训练一致，无失配。
  • injector line 400 注释"training dataset's projected_gravity is identically [0,0,0]"是旧数据集的过时残留，应删（非功能 bug，但误导）。
→ "填 gravity = 高 ROI" 已是既成事实；gravity 通道（编码 roll/pitch 倾斜）已在用。注意它仍不测 yaw、不解 static onset。

8. 修订后的执行顺序 Revised order (post-review)

  1. [已做] 审计 gravity + state 关节序：gravity 真值已对齐；state 序需在 B5 一并核。
  2. re-anchor baseline（deploy）：切 fragile 动作前校验 pose 距离/gravity，把 robot 带回已演示站立姿再发 prompt。最高 ROI、零训练风险。
  3. Tier-3 onset priming（deploy）：注入短的、动作专属、验证安全的 onset 前缀 token，再交还 VLA。冷切换的针对性疗法。
  4. velocity + T=2 history 重训（本方案③）：处理 B1–B5 后做。T=2 先于 T=3（60s 高相关数据，T=3 主要增记忆/过拟合）。验收 = settle-switch + drift 复现，非 OL-MSE。

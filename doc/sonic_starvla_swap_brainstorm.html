LeSONIC · 架构侧 / Architecture route

StarVLA-Qwen3.5-4B-PI_v3 能否替换 GR00T N1.7 做 SONIC 全身动作?

Can StarVLA replace GR00T as the SONIC token producer? — 三模型联合头脑风暴 (Opus 4.8 + codex gpt-5.5 + mimo-v2.5-pro) · 2026-06-09
TL;DR — 接口上能换,但这是一次横向移动 (lateral move),不是修复。三模型独立结论一致:替换骨干修不了已知 3 个缺陷里的任何一个,因为真瓶颈在 动作头的离散损失(把 FSQ 离散码当连续回归)和 观测空间缺记忆/相位,这两者都与 VLM 骨干无关。
You can swap it in, but it's a lateral move: it fixes 0 of the 3 known flaws because the real bottleneck is the head's loss (continuous regression on a discrete FSQ target) and the memory-less observation space — both orthogonal to the VLM backbone.

裁决 / Verdict:仅当把它当作 ① 干净的 A/B 基线(同一冻结 WBC 下对比 GR00T vs StarVLA)或 ② 顺手实现 FSQ-aware 损失的载体,才值得做。不要把"换骨干"本身当成能力提升。mimo 更激进:直接 skip,先在 GR00T 上修 head + obs。

1 · 问题 / The question

我们已经训过 StarVLA-Qwen3.5-4B-PI_v3-PickOrange(SO-101 单臂抓橙子,strict 20-round = 35%)。自然的想法:既然有这套 StarVLA 训练/serving 基建,能不能把同一个 PI_v3 模型训成 SONIC 的 token 生产者,替掉 GR00T N1.7,喂同一个冻结 GEAR-SONIC WBC,做出一样的串联动作(蹲→走→踢→舞,循环)?

SONIC 管线天然分成两半:可替换的 token 生产者 + 固定的 冻结 WBC 解码器。GR00T 只是当前的 token 生产者。任何能在正确节奏吐出合法 64 维 FSQ token 的模型都能接线驱动 decoder —— 这正是注入器 hook 的设计(actor_module.decode("g1_dyn", {token_flattened, proprioception}) 被外部 token 覆盖)。但"合法 token"是必要非充分:序列时序、相位、proprio 兼容性、token 是否落在 WBC latent 流形上,决定动作稳不稳。所以"接口能不能接"答案是能,且 trivial。真问题是换了有没有用。

3 · 输出空间:同一个连续回归原罪 / Same continuous-regression sin

这是所有人都认定的核心盲点。GR00T 和 StarVLA-PI_v3 都是 flow-matching 连续回归头。把 GR00T 换成 PI_v3 的头,不是对 FSQ 离散目标的修复 —— 是同一个把离散量化码当连续欧氏回归的罪。

mimo 直接翻了源码坐实:

# StarVLA flow-matching 头 (LayerwiseFM_ActionHeader.py / GR00T_ActionHeader.py)
loss = ((pred_actions - velocity) ** 2).mean()   # MSE-on-velocity,连续目标
# predict_action():从高斯噪声 Euler 积分去噪 → 输出连续值,落在 FSQ 网格点【之间】
# GR00T N1.7 的 DiT flow 头同款连续回归(各自 action_head 路径,非同一文件)

注:行号随版本漂移,此处只标文件;上面是 StarVLA 的 head,GR00T N1.7 是各自的 action_head 路径,但损失形式同款连续回归。

flow-matching 的多峰性帮不上忙:这份监督数据对给定 (prompt, proprio) 每个动作只录了一条 token 轨迹 —— 即便真实动作存在相位/步幅/微风格上的多峰,数据里未被观测到,也不是瓶颈。多峰能力(如"伸左手 OR 伸右手")在这里无从发挥。codex 还补了一刀:落在网格点之间未必意味"中间动作",更可能是 decoder 没训练过的 latent garbage,反而比 GR00T 漂得更狠。

所以 flaw #2 真正的修法与骨干无关:① 每维 交叉熵(对观测到的 snap 网格值 bin 分类;若抽到 SONIC 原生 quantizer 索引则可能是 32 级 FSQ,不是 17)② 预测 pre-quant 连续 latent 再过 SONIC 自己的 FSQ quantizer。这两刀在 GR00T 上一样能做 —— StarVLA 在这点上没有特权。我们现有的推理侧 snap-to-grid(SONIC_SNAP_GRID)对两个骨干同样适用。
The real fix for flaw #2 is a loss change (per-dim CE / pre-quant latent), doable on GR00T too — StarVLA is not privileged here.

6 · 视觉浪费 + 语言 grounding / Vision wasted

ego 相机 attach 到 pelvis 朝下看地面,对动作判别视觉信号大概率 ≈ 0(需 vision-off / proprio-only ablation 才是实测断言,目前是强先验)—— prompt + proprio 应驱动一切。果真如此,Qwen3-VL 更强的视觉编码器在"每个动作都看同一片灰地面"时用武之地极小,甚至(codex)在光照/地面纹理变化时引入无关 nuisance 特征。

语言侧呢?Qwen3-VL-4B 确实比 Cosmos-Reason2-2B 强,但 prompt 空间只有 7 个固定串("do a squat" / "perform macarena" …),本质是 7-way 分类。Cosmos-2B 的语言 grounding 已经够。只有当 SONIC scale 到几百个多样自然语言 prompt(改写、组合、"squat then kick while waving")时,Qwen3-VL 的语言能力才值钱 —— 那是未来的数据-scaling 论点,不是今天的架构论点。

8 · 盲点 / Blind spots

  1. codex 非-VLA baseline 拷问:既然视觉≈0,先跑一个极小非 VLA baseline ——
    prompt-ID + proprio (+phase) → MLP/Transformer → 40×64 token chunk。
    如果它打平 StarVLA,那这次换骨干就只是基建工作、不是模型进步。这是必须先排除的对照。
  2. codex scripted scaffold 拷问:串联序列能成,主要是时间线注入器 + bootstrap token 的功劳,不是学到了"串联"。策略只是在一个外部脚手架(换 prompt + 喂 bootstrap)里填局部 chunk。换骨干不改变这点。
  3. mimo 层次错位拷问:你在优化骨干 (VLM),而瓶颈在 头(离散目标上的连续回归)+ obs 空间(无记忆)。换 VLM 是最贵、对真实失败模式影响最小的改动。先修便宜的。
  4. codex 真正的接口不是"VLA 控制机器人",而是 "VLA 为冻结 WBC 挑选 latent 参考"。核心科研问题不是"Qwen3-VL 是否更会机器人",而是 "你的 token 分布是不是 SONIC decoder 的忠实控制语言" —— 离散码保真度、相位可观测性、WBC latent 流形覆盖率,都比 VLM 大小重要。
  5. codex WBC 可行性在 VLA 上游:即便 token 预测完美,WBC 也执行不了它冻结 decoder 稳定流形之外的动作。判 StarVLA 之前要先过 严格终止(RELAX 关)下的 WBC 跟踪筛查 —— 这道闸门与换不换骨干无关,是动作本身能不能被这个 WBC 跟住。
  6. codex chunk 边界 / 相位重置:horizon-40 预测可能 MSE 好看,却在段落起点 / prompt 切换处崩(无相位/历史)。评测指标必须含 prompt 切换后头 N 步误差 + chunk 边界不连续度,不能只看整段 MSE。

9 · 最便宜的 go/no-go 实验 / Cheapest experiment

三方一致:别从 live Isaac rollout 起步。先离线 token 生产者测试。

  1. 实验 A — 离线过拟合 sanity:复用现成 wsagi/SONIC-VLA-BonesSeed,写 UNITREE_G1_SONIC data_config(action=64 token / video=ego_view / state=29+gravity),冻 VLM 训 PI_v3 头,horizon=40,显式 token 归一化,推理 snap-to-grid,跑 ~500-2000 步。报告不只 MSE:连续 MSE + snapped MSE + 每维 FSQ bin 准确率 + off-grid 比例 + 出界比例 + 每动作 + chunk 边界不连续度 + prompt 切换后头 N 步误差。
  2. 实验 B — 真正的 held-out:7×1ep 下帧级随机切分是假的。用 LOMO(留 1 动作整 ep)/ 每动作留后半段 / prompt 改写 / transition 边界。
  3. 实验 C — 最小 Isaac rollout:先 squat→dance→macarena(自持,免 bootstrap),再 squat→walk→kick→dance(用与 GR00T 同款 bootstrap 拐杖)。
预测(三方一致):它会学会(loss 降、MSE 破 per-motion-mean 基线 0.039),因为 7-way 记忆很容易。但这什么也没证明 —— go/no-go 信号不是"它能不能学",而是"它学得比 GR00T 好吗"。先验强烈倾向"否"(同一原罪 + 更差的预训练 base + 拉伸的 horizon + 浪费的视觉),但只有真跑 A/B 才能证。

10 · 裁决 + 正确的下一步 / Verdict & what to do instead

❌ 按"换骨干求提升"来做 = 不值得。它修 0/3 缺陷、浪费 Qwen3-VL 视觉、放弃 GR00T 预训练 base、拉伸 horizon 2.5×,只换来 ~200 行代码统一,却把同样 3 个缺陷换个骨干名字写进 model card。
✅ 真要做,只在这两种 framing 下值得:
  1. 干净的 A/B 基线:同一冻结 WBC + 同一数据下,StarVLA 若打平 GR00T → 科学上有用的负面结果(证明瓶颈是数据+obs,不是模型家族,堵死"换更好的 VLA 就行"的幻想)。诚实结论只能是 "StarVLA 可作 SONIC 冻结 WBC 的 drop-in 连续 token 生产者",不能说 "StarVLA 提升了 prompt 驱动全身控制"。
  2. 修 flaw #2 的载体:如果换骨干的同时立刻改成 per-dim CE / pre-quant latent 损失,才有增量价值。

更该先做的(按 ROI 排序,全在 GR00T 上,零骨干改动):

  1. 先修 flaw #2 —— per-dim 交叉熵 / pre-quant latent,只重训头。P0 roadmap 已列。
  2. 再修 flaw #3 —— obs history + phase clock,只改 obs 空间。
  3. 然后才谈骨干 A/B —— 等有了能真正压榨头容量的干净 baseline,再有意义地对比 Cosmos-2B vs Qwen3-VL。

一个例外:若计划把 SONIC scale 到几百个多样自然语言 prompt(组合 / 改写 / "squat then kick while waving"),则 Qwen3-VL 的语言 grounding 变得有价值 —— 但那是未来数据-scaling 论点,不是今天的架构论点。

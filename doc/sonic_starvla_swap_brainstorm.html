LeSONIC · 架构侧 / Architecture route

StarVLA-Qwen3.5-4B-PI_v3 能否替换 GR00T N1.7 做 SONIC 全身动作?

Can StarVLA replace GR00T as the SONIC token producer? — 三模型联合头脑风暴 (Opus 4.8 + codex gpt-5.5 + mimo-v2.5-pro) · 2026-06-09
TL;DR — 接口上能换,但这是一次横向移动 (lateral move),不是修复。三模型独立结论一致:替换骨干修不了已知 3 个缺陷里的任何一个,因为真瓶颈在 动作头的离散损失(把 FSQ 离散码当连续回归)和 观测空间缺记忆/相位,这两者都与 VLM 骨干无关。
You can swap it in, but it's a lateral move: it fixes 0 of the 3 known flaws because the real bottleneck is the head's loss (continuous regression on a discrete FSQ target) and the memory-less observation space — both orthogonal to the VLM backbone.

裁决 / Verdict:仅当把它当作 ① 干净的 A/B 基线(同一冻结 WBC 下对比 GR00T vs StarVLA)或 ② 顺手实现 FSQ-aware 损失的载体,才值得做。不要把"换骨干"本身当成能力提升。mimo 更激进:直接 skip,先在 GR00T 上修 head + obs。

1 · 问题 / The question

我们已经训过 StarVLA-Qwen3.5-4B-PI_v3-PickOrange(SO-101 单臂抓橙子,strict 20-round = 35%)。自然的想法:既然有这套 StarVLA 训练/serving 基建,能不能把同一个 PI_v3 模型训成 SONIC 的 token 生产者,替掉 GR00T N1.7,喂同一个冻结 GEAR-SONIC WBC,做出一样的串联动作(蹲→走→踢→舞,循环)?

SONIC 管线天然分成两半:可替换的 token 生产者 + 固定的 冻结 WBC 解码器。GR00T 只是当前的 token 生产者。任何能在正确节奏吐出合法 64 维 FSQ token 的模型都能接线驱动 decoder —— 这正是注入器 hook 的设计(actor_module.decode("g1_dyn", {token_flattened, proprioception}) 被外部 token 覆盖)。但"合法 token"是必要非充分:序列时序、相位、proprio 兼容性、token 是否落在 WBC latent 流形上,决定动作稳不稳。所以"接口能不能接"答案是能,且 trivial。真问题是换了有没有用。

3 · 输出空间:同一个连续回归原罪 / Same continuous-regression sin

这是所有人都认定的核心盲点。GR00T 和 StarVLA-PI_v3 都是 flow-matching 连续回归头。把 GR00T 换成 PI_v3 的头,不是对 FSQ 离散目标的修复 —— 是同一个把离散量化码当连续欧氏回归的罪。

mimo 直接翻了源码坐实:

# StarVLA flow-matching 头 (LayerwiseFM_ActionHeader.py / GR00T_ActionHeader.py)
loss = ((pred_actions - velocity) ** 2).mean()   # MSE-on-velocity,连续目标
# predict_action():从高斯噪声 Euler 积分去噪 → 输出连续值,落在 FSQ 网格点【之间】
# GR00T N1.7 的 DiT flow 头同款连续回归(各自 action_head 路径,非同一文件)

注:行号随版本漂移,此处只标文件;上面是 StarVLA 的 head,GR00T N1.7 是各自的 action_head 路径,但损失形式同款连续回归。

flow-matching 的多峰性帮不上忙:这份监督数据对给定 (prompt, proprio) 每个动作只录了一条 token 轨迹 —— 即便真实动作存在相位/步幅/微风格上的多峰,数据里未被观测到,也不是瓶颈。多峰能力(如"伸左手 OR 伸右手")在这里无从发挥。codex 还补了一刀:落在网格点之间未必意味"中间动作",更可能是 decoder 没训练过的 latent garbage,反而比 GR00T 漂得更狠。

所以 flaw #2 真正的修法与骨干无关:① 每维 交叉熵(对观测到的 snap 网格值 bin 分类;若抽到 SONIC 原生 quantizer 索引则可能是 32 级 FSQ,不是 17)② 预测 pre-quant 连续 latent 再过 SONIC 自己的 FSQ quantizer。这两刀在 GR00T 上一样能做 —— StarVLA 在这点上没有特权。我们现有的推理侧 snap-to-grid(SONIC_SNAP_GRID)对两个骨干同样适用。
The real fix for flaw #2 is a loss change (per-dim CE / pre-quant latent), doable on GR00T too — StarVLA is not privileged here.

6 · 视觉浪费 + 语言 grounding / Vision wasted

ego 相机 attach 到 pelvis 朝下看地面,对动作判别视觉信号大概率 ≈ 0(需 vision-off / proprio-only ablation 才是实测断言,目前是强先验)—— prompt + proprio 应驱动一切。果真如此,Qwen3-VL 更强的视觉编码器在"每个动作都看同一片灰地面"时用武之地极小,甚至(codex)在光照/地面纹理变化时引入无关 nuisance 特征。

语言侧呢?Qwen3-VL-4B 确实比 Cosmos-Reason2-2B 强,但 prompt 空间只有 7 个固定串("do a squat" / "perform macarena" …),本质是 7-way 分类。Cosmos-2B 的语言 grounding 已经够。只有当 SONIC scale 到几百个多样自然语言 prompt(改写、组合、"squat then kick while waving")时,Qwen3-VL 的语言能力才值钱 —— 那是未来的数据-scaling 论点,不是今天的架构论点。

8 · 盲点 / Blind spots

  1. codex 非-VLA baseline 拷问:既然视觉≈0,先跑一个极小非 VLA baseline ——
    prompt-ID + proprio (+phase) → MLP/Transformer → 40×64 token chunk。
    如果它打平 StarVLA,那这次换骨干就只是基建工作、不是模型进步。这是必须先排除的对照。
  2. codex scripted scaffold 拷问:串联序列能成,主要是时间线注入器 + bootstrap token 的功劳,不是学到了"串联"。策略只是在一个外部脚手架(换 prompt + 喂 bootstrap)里填局部 chunk。换骨干不改变这点。
  3. mimo 层次错位拷问:你在优化骨干 (VLM),而瓶颈在 头(离散目标上的连续回归)+ obs 空间(无记忆)。换 VLM 是最贵、对真实失败模式影响最小的改动。先修便宜的。
  4. codex 真正的接口不是"VLA 控制机器人",而是 "VLA 为冻结 WBC 挑选 latent 参考"。核心科研问题不是"Qwen3-VL 是否更会机器人",而是 "你的 token 分布是不是 SONIC decoder 的忠实控制语言" —— 离散码保真度、相位可观测性、WBC latent 流形覆盖率,都比 VLM 大小重要。
  5. codex WBC 可行性在 VLA 上游:即便 token 预测完美,WBC 也执行不了它冻结 decoder 稳定流形之外的动作。判 StarVLA 之前要先过 严格终止(RELAX 关)下的 WBC 跟踪筛查 —— 这道闸门与换不换骨干无关,是动作本身能不能被这个 WBC 跟住。
  6. codex chunk 边界 / 相位重置:horizon-40 预测可能 MSE 好看,却在段落起点 / prompt 切换处崩(无相位/历史)。评测指标必须含 prompt 切换后头 N 步误差 + chunk 边界不连续度,不能只看整段 MSE。
  7. Opus 复审补 §5 按 serving 栈写,但 A/B 根本不需要 server:GR00T 侧的对照数字来自离线 dump 管线(数据集 obs → predict_action → npz → 注入器),StarVLA 同理 —— 直接调 framework.predict_action() 拿 normalized_actions,不经 websocket / StarVLAServicePolicyClient / PolicyServerWrapper 的反归一化。§5 的"归一化陷阱"从一张表收敛为训练侧一个选择(identity:action key 不进 normalization_modes,loader 源码 if modality_key in self.normalization_modes 不在即跳过),dump 侧拿到的就是 token 原值。serving 那张表只在 A/B 之后做 live demo 才生效。
  8. Opus 复审补 state 维度写错 + 两个 state 真坑:§5 说"29-DoF + gravity = 32 维" —— 错,实录数据集是 observation.state=43(含手指,后 14 维零填) + projected_gravity=3 = 46 维。更要命:① PI_v3 state2str_transform 的离散化 bins 固定 [-1,1](256 级) —— 关节弧度(膝/肩 >2 rad)不先 min_max 归一化直接喂,出界值全 clip 进边缘 bin,proprio 信号被毁(文档归一化一节只盯 action 漏了 state);② loader 的 state 打包受 datasets.vla_data.include_state 控制,默认 off(SO101 就是 stateless 跑的) —— 忘开就退化成 (image,prompt)→token,与吃 proprio 的 GR00T 不可比。
  9. Opus 复审补 teacher-forcing 开环口径要写死:离线 dump 喂的是数据集真 obs(teacher forcing)。GR00T 锚点同口径 → A/B 公平,但结论只能限定开环 token 保真度;闭环漂移 / 一次性动作 settle 到站立,是 live 才测得到的另一层。
  10. Opus 复审补 公平性按 epoch(样本量)对齐,不按 step,且 LR schedule 必须铺满:GR00T flow3 = 6000 步 × global batch 4 = 24k 样本 ≈ 4.15 epoch / 5777 帧。StarVLA bs=4 × 6000 步同样本量。教训:早前 GR00T "0.065 平台"纯粹是 max_steps 截断让 cosine LR 早衰的欠训假象 —— 别在 StarVLA 上重演后错判"StarVLA 不行"。

9 · 最便宜的 go/no-go 实验 / Cheapest experiment

三方一致:别从 live Isaac rollout 起步。先离线 token 生产者测试。

2026-06-10 修订:测床换成 LAFAN flow3,不是 BonesSeed。GR00T N1.7 刚在 datasets/sonic_vla_lerobot_flow3(8 窗口 / 5777 帧 / 8 prompt)上训完,锚点新鲜: train loss 0.052 · open-loop MSE 平均 0.00257 · 全 8 窗口 ≤0.004(checkpoint-6000, 6000 步 × global batch 4)。同数据 + 同样本量训 StarVLA 才是干净 A/B;BonesSeed 的 GR00T 数字 (0.0011)是另一口径的 8k 老 run,扯进来只会浑水。 Testbed switched to LAFAN flow3 — the GR00T anchor numbers are fresh and epoch-matched comparison is possible.
  1. 实验 A — 离线过拟合 sanity(LAFAN flow3):写 unitree_g1_sonic data_config(action=64 token / video=ego_view / state=43+gravity=46,include_state: true + state min_max),冻 VLM 训 PI_v3 头,horizon=40,action 走 identity(不进 normalization_modes),推理 snap-to-grid,先 ~500 步 smoke 再放量。报告不只 MSE:连续 MSE + snapped MSE + 每维 FSQ bin 准确率 + off-grid 比例 + 出界比例 + 每窗口 + chunk 边界不连续度 + prompt 切换后头 N 步误差。
  2. 实验 B — 真正的 held-out:7×1ep 下帧级随机切分是假的。用 LOMO(留 1 动作整 ep)/ 每动作留后半段 / prompt 改写 / transition 边界。
  3. 实验 C — 最小 Isaac rollout:先 squat→dance→macarena(自持,免 bootstrap),再 squat→walk→kick→dance(用与 GR00T 同款 bootstrap 拐杖)。
预测(三方一致):它会学会(loss 降、MSE 破 per-motion-mean 基线 0.039),因为 7-way 记忆很容易。但这什么也没证明 —— go/no-go 信号不是"它能不能学",而是"它学得比 GR00T 好吗"。先验强烈倾向"否"(同一原罪 + 更差的预训练 base + 拉伸的 horizon + 浪费的视觉),但只有真跑 A/B 才能证。

10 · 裁决 + 正确的下一步 / Verdict & what to do instead

❌ 按"换骨干求提升"来做 = 不值得。它修 0/3 缺陷、浪费 Qwen3-VL 视觉、放弃 GR00T 预训练 base、拉伸 horizon 2.5×,只换来 ~200 行代码统一,却把同样 3 个缺陷换个骨干名字写进 model card。
✅ 真要做,只在这两种 framing 下值得:
  1. 干净的 A/B 基线:同一冻结 WBC + 同一数据下,StarVLA 若打平 GR00T → 科学上有用的负面结果(证明瓶颈是数据+obs,不是模型家族,堵死"换更好的 VLA 就行"的幻想)。诚实结论只能是 "StarVLA 可作 SONIC 冻结 WBC 的 drop-in 连续 token 生产者",不能说 "StarVLA 提升了 prompt 驱动全身控制"。
  2. 修 flaw #2 的载体:如果换骨干的同时立刻改成 per-dim CE / pre-quant latent 损失,才有增量价值。

更该先做的(按 ROI 排序,全在 GR00T 上,零骨干改动):

  1. 先修 flaw #2 —— per-dim 交叉熵 / pre-quant latent,只重训头。P0 roadmap 已列。
  2. 再修 flaw #3 —— obs history + phase clock,只改 obs 空间。
  3. 然后才谈骨干 A/B —— 等有了能真正压榨头容量的干净 baseline,再有意义地对比 Cosmos-2B vs Qwen3-VL。

一个例外:若计划把 SONIC scale 到几百个多样自然语言 prompt(组合 / 改写 / "squat then kick while waving"),则 Qwen3-VL 的语言 grounding 变得有价值 —— 但那是未来数据-scaling 论点,不是今天的架构论点。

11.5 · 第③④刀放量 = 双负结果:masked CE + proprio history 全线退步 / P0 series, negative at scale

用户裁决 CE v1 live demo「动作幅度很小,应该还需要改进」后,按 三模型 roadmap 的 P0(改 loss/建联合分布)→ P1(治记忆/加 history)正序,实施了两刀。两刀在开环 teacher-forcing 保真度上都是负结果, 且比 §11.4 的 layerwise 更狠 —— 连「每窗输出常数均值」的模板基线都打不过。 Both the MaskGIT joint-distribution head and proprio history regressed at 8-window scale — worse than a constant per-window-mean template, the weakest possible baseline.

裁决:不是 serve bug,不是欠训。开环 dump 走训练 dataloader、绕过整个 serving 栈,数字本身就比 CE v1 差 2.5×;5000→6000 步已平台。 「加 head 表达力 / 建联合分布」这条路在本数据规模上实测退步 —— 但 mimo 复审指出还有 d_model 不公(1024 vs 768)与 train/inference gap 两个 confound 未排, 「架构死刑」需两个 ~1 天消融坐实(见本节末「诚实边界」)。

为什么 masked 联合分布在这里是错的 / Why the joint head is wrong here

  1. 数据撑不起联合分布(主因)。数据集总信息量 ≈ 5777 帧 × 64 维 × 3.61 bit/维(实测每维 token 边际熵)≈ 1.33 Mbit; masked head 58.8M 参数,每个 prompt 仅 ~720 帧。MaskGIT 建 p(masked | revealed, context) over 3120 位置的前提是 大量多样序列让联合统计可估;8 条轨迹下,3120-位置联合分布是欠定的,过参数化 head 退化到比逐维均值还差。 小数据下 per-dim 独立 CE 的强归纳偏置不是缺陷,是对的先验 —— 联合建模在此是有害的自由度。 Joint modeling needs data the 8-window set doesn't have; the independence prior of per-dim CE is correct here, not a defect.
  2. 幅度塌缩 = confidence-ranking 揭示调度引起,非 head 权重坏(两个控制实验隔离)。 K=10 confidence 揭示下 token std 0.147 = GT 0.216 的 68%(CE v1 argmax 0.219=GT)。机制:confidence-first 先揭示最自信=低熵「接近静止」维(k=0 占数据 12%), 锁定后剩余维被逼「与静止联合一致」→ 整序列拉向静止。两个控制实验各自隔离出元凶:
    解码(同 masked ckpt)token std(GT 0.216)MSE64bin_acc
    confidence-rank, K=10(默认)0.147(68% GT)0.043717.7%
    SONIC_CE_MASK_STEPS=1 单次前向0.19(88% GT)0.048420.6%
    SONIC_CE_REVEAL=random 随机揭示0.18(83% GT)0.044818.1%
    结论(经 mimo 复审修正):去掉 confidence 排序(随机揭示或单次前向)幅度即恢复到 83-88% GT —— 塌缩是调度的产物,不是 MaskGIT 范式的死刑,我先前判「固有病」过早。 但三种解码 MSE64 全在 0.044-0.048,一律远差于 CE v1 的 0.0174 → 即便修掉塌缩,masked head 在 5777 帧上依然输给 per-dim 独立 CE。幅度塌缩是用户「动作幅度小」的来源之一,可缓解;但 head 的保真劣势不可缓解。 Controls (per mimo's review): dropping the confidence ranking restores std to 83-88% GT — the collapse is the schedule, not the MaskGIT paradigm; my "inherent" call was premature. But MSE64 stays 0.044-0.048 across all decodes, far above CE v1's 0.0174 — fixing the collapse doesn't fix the head's fidelity loss on this data size.
  3. history 经「离散化塞文本」通路是错的工程。K=3 帧 normalized state 各过 256-bin np.digitize 成数字串, 拼成 [STATE_HIST] … [ACTION] 进 instruction。实测相邻帧约 48% 的(维,帧)动不到 1 个 bin —— 近半历史 token 与当前帧同 bin(半冗余),却把文本加长 ~138 token,frozen VLM 无从学会忽略 → 净稀释 cross-attn memory。 history 让每一窗都更差,坐实是通路实现错,不是「history 概念无用」。roadmap 把 history 排在 loss 之后没错, 错在叠在一个已经塌缩的 masked head 上 + 用文本通道而非数值 state encoder。 Half the history tokens are bin-identical to the current frame (redundant) yet add ~138 tokens that dilute the cross-attn memory — wrong pathway, not a wrong idea.
主结论:任何「加 head 容量/表达力」的改动(layerwise / masked / history),在 5777 帧上一律开环退步。 最可能的瓶颈是数据规模而非 head 表达力,方向应转 roadmap P2 扩数据。 Main finding: every capacity-adding head change regresses open-loop on 5777 frames; the likely bottleneck is data scale, pivot to P2.
⚠️ 诚实边界 / 待排 confound(mimo 复审 2026-06-11,采纳)—— 「架构死刑」尚未完全证实,两个更便宜的解释未排除:
  1. d_model 不公平对比(必须重训才能排)。CE v1 用 d_model=1024,masked 用 768(当初为省 3120-token self-attn 显存而降),每位置表达力差 (1024/768)²≈1.78×。 这是被当成常量的 confound。判「小数据撑不起联合分布」前,必须先排除「768 不够用」这个更简单的解释。待办实验 A(~1 天):masked + d_model=1024 + 随机揭示 + K=20,同 6000 步;仍输 CE v1 → 架构死刑坐实,接近 → 是 d_model/调度。
  2. train/inference gap(masked 路线固有,但可缓解)。训练时 revealed 位置永远喂 36% 完美 GT token;推理时全 [MASK] 起步、喂自己的预测,误差逐轮累积。33-bin 信息密度低,误差传播比图像 MaskGIT 更猛。随机揭示已部分缓解(见上),但根治需 scheduled-sampling 式训练。
  3. history 概念未被否定,只否定了文本通路。「离散化塞文本」破坏 state 维间物理耦合 + 稀释 cross-attn,确实是错通路;但连续 state encoder(K×46 → MLP/1D-conv → concat 进 head cross-attn memory)是完全不同的信息瓶颈,未测。 待办实验 B(~1 天):CE v1 + 数值 state encoder,有提升 → history 概念有效、文本通路是 bug。
裁决:先做 A/B 两个 ~1 天的便宜消融排除 confound,再决定是否投 5-10 天的 P2 扩数据。不在排除 confound 前直接跳最贵方案。 Per mimo: the d_model mismatch (1024 vs 768) and the train/inference gap are unexcluded confounds; run the two ~1-day ablations (A: d_model=1024 + random reveal; B: CE v1 + continuous state encoder) before committing to data scaling.

11.6 · 附带修掉一个真 serve bug:state 归一化的置换错位 / A real serving bug, fixed

排查上面负结果时,顺手验证了 serving 端 state 归一化与训练侧是否逐元素对齐 —— 发现一个独立的真 bug(与开环负结果无关, 因为 dump 走训练 dataloader 不过 serve,但它污染了所有 live demo 的幅度判断)。

根因:concat-then-normalize 双重置换。serve_starvla_sonic.py 的 MinMax 把 injector 送来的 「按 _STATE_GROUPS 顺序拼接的 state43」直接当 raw observation.state 列做归一化。但 _STATE_SLICES 是个置换: raw 列里 left_hand=22:29 排在 right_arm=29:36 之前,与 _STATE_GROUPS 的列举顺序相反。 结果 right_arm 通道被喂进 left_hand 的 min/max、left_hand 被喂进 right_arm 的退化(全零)统计 → right_arm proprio 通道在 serve 时恒为常数。 _STATE_SLICES is a permutation; concat-then-normalize fed right_arm through the wrong (degenerate, all-zero) stats — the channel came out constant.

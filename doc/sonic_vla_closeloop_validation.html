Close-the-loop validation — GR00T in-loop drives the SONIC whole-body controller. 实时 prompt / 泛化 / 纠偏 / 扩数据，按依赖与数据约束排序。

TL;DR. derisk（§10）已证 GR00T 能把动作压进 64 维 token、WBC 兜平衡——但全是开环回放（离线 dump token 再注入），观测漂移下到底稳不稳没验过。本计划把"实时 prompt 活循环 / held-out 泛化 / 多 clip+RSI / 闭环纠偏"四件事按依赖关系重排：闭环活循环是地基，泛化/纠偏都长在它上面；held-out 新动作与多 clip 被数据卡死（需 HF 30G），降级为可选或弱化版。 derisk proved the token bottleneck works, but only open-loop. The real untested risk is drift robustness. Closed-loop is the foundation; generalization/correction build on it; new-motion held-out is data-blocked.

1. 为什么闭环是地基

开环回放（当前 §10 的注入）每步喂的是离线 dump 好的 token 序列，env 的真实观测不参与。所以它测的是「GR00T 在录制轨迹的观测上能否复现 token」——本质是记忆。三件真问题它都测不到：

真问题开环回放能测吗为什么只有闭环能测
实时 prompt 当场动❌开环 token 是离线算好的；闭环才是「env 当前状态 → GR00T 现算 token」
观测漂移鲁棒性❌WBC 解码有微小误差 → 真实轨迹偏离录制轨迹 → GR00T 看到没 dump 过的观测，必须现场泛化
长动作累积漂移 / 自纠❌开环误差只会累积；闭环里 GR00T 持续读真 obs，理论上能自纠——但要验
关键洞察。 闭环本身就是一种 held-out：WBC 解码误差让机器人进入录制时没出现过的状态，GR00T 必须对这些漂移状态出好 token。所以「闭环单动作不摔」已经是比开环强得多的泛化证据，不需要新动作数据。Closed-loop is itself a held-out test — the controller drifts into unseen states the model must handle, no new-motion data required.

2. 闭环桥架构：双 venv + ZMQ

硬约束：GR00T N1.7 跑 transformers 4.57.3 的 venv，SONIC eval 跑 Isaac conda——不能同进程。所以走 GR00T 自带的 ZMQ server/client（run_gr00t_server.py + PolicyClient，wire 协议在记忆 gr00t-n17-leisaac-wire-debug 早已 debug）。

三个工程难点（doc 会随实测更新踩坑）：① obs 模态对齐（ego cam 480×640 + state 含 projected_gravity + prompt，必须和训练完全一致）；② 动作分块（GR00T 出 40，SONIC 每步要 1 → 缓冲）；③ 延迟（GR00T 推理 ~100–400ms，40 步一次摊薄，但首步会卡）。

4. 数据约束：卡死的与降级方案

硬约束。 本地 demo_robot_filtered.pkl 只含 7 条训练动作。真正的「新动作泛化」(P3b) 和「多 clip 扩数据」(P5) 都要从 HF 下 ~30G 的多动作 robot_filtered。
  • 降级 P3a 替代 P3b：用 init 扰动 / 起始相位偏移制造「没录过的状态」，在闭环里测泛化——不需要新动作数据，且闭环本身已是 held-out（见 §1 洞察）。
  • P3b/P5 转可选：跑完 P1–P4 拿到结论后，再决定是否值得为「新编舞动作」付 30G 下载 + 重训。届时单独问。

踩坑存档（reusable）：
  • int('True') 崩溃根因：IsaacLab app_launcher.py:561 读环境变量 HEADLESS 做 int(os.environ["HEADLESS"]) → 用 HEADLESS=True bash script 调用时 'True' 泄漏给 int() 崩。修：脚本读完即 unset HEADLESS，只用 hydra +headless= arg 控制。
  • headless 下 debug_vis=True 报错：绿骨架 viz 需 viewer；headless 只在 GUI 时加 debug_vis。
  • pkill -f 'eval_agent' 自杀：当前 shell 命令行含该字串被自己匹配 → 按 PID 过滤排除 $$。
  • 注入 token 来自只读 msgpack buffer → torch.as_tensor warn；.copy() 消除。
2026-06-04 · P1 关键发现：闭环 GR00T 正确但「一次性动作卡在静止」。 用户肉眼：kick 闭环站着微晃、不踢。逐层诊断（reusable 方法论）：
  • obs 没喂错：live state 与录制帧 L2=0.053（匹配 kick 相位 t≈160），nonzero 模式一致；camera mean 102.8 ≈ 录制 103.4（不是黑屏）。
  • GR00T 没坏：live token[0] 最匹配 dump phase t=133（L2=0.147），远离 dump[0]（0.733）→ GR00T 对「观测到的相位」正确出 token。
  • 根因 = 无记忆 obs 条件 vs 一次性动作：GR00T 单帧（delta_indices=[0]）只看当前姿态。kick 起止都是站立姿→ 策略无法区分「将踢」与「踢完」→ 驱到静止就停。开环强喂 token[0..164] 没这问题。
2026-06-04 · 周期 vs 一次性 对照实验（决定性）。 同一闭环、headless、各 ~110s：
动作类型dpose（关节位移/查询）tok_std闭环行为
kick一次性→静止0.46→0.10→0.06（衰减冻结）0.16（平）❌ 卡在站立
macarena周期舞0.38→1.96→1.91→2.31（持续）0.27（高）✅ 自持跳舞
结论：闭环架构对周期动作自持、对一次性回静止动作失效。这是无记忆策略的本质局限，非 bug。Closed-loop self-sustains cyclic motions but stalls one-shot return-to-rest motions — inherent to a memoryless obs-conditioned policy, not a bug.
清单调整（按发现）：
  • P2 重定义：不再是「7 个都跑闭环」，而是验证周期/一次性的切分——预期 walk/dance/macarena ✅ 自持，kick/squat/jump/lunge ❌ 卡静止（jump 看重复性）。
  • 新 P-fix（一次性动作的闭环解法，需评估）：① 训练加时间上下文（delta_indices 含历史帧 → 区分将踢/踢完，需重训）；② obs 加相位/进度信号；③ 触发式：检测「prompt 切换/外部触发」时强制从 phase 0 喂一段开环 token 引导进入动作，再交回闭环。
  • held-out / 多clip 仍数据卡（30G HF）。
2026-06-04 · P2 全 7 动作扫描 → 修正上面的「周期/一次性」二分（如实）。 把全部 7 条都闭环跑一遍后，干净的二分不成立——squat/lunge 也回到站立却照样自持。真实经验格局：
动作（时长）dpose 收敛闭环判定

修正结论：闭环对 4–5/7 动作自持，仅 kick 和 walk 卡死。判别因素不是严格周期性，而是「机器人是否陷入策略逃不出的通用站立静止姿」——kick 是唯一最短的单次事件（3.3s），walk 末尾刻意停步；其余动作（含同样回到站立的 squat/lunge）在片段内保持足够动态/重复，闭环不掉。 Full 7-motion sweep walks back the clean cyclic/one-shot split: closed-loop sustains 4–5/7; only kick (uniquely short single event) and walk (deliberate stop) stall. The real discriminator is whether the robot settles into a generic standing rest pose the memoryless policy can't escape.
方法论存档（reusable）：① 单 GPU 连跑多个 Isaac eval 时，timeout 杀 conda wrapper 但 python 子进程存活成 orphan 占 GPU，累积 4 个就把 24G 撑爆（create_articulation_view: NoneType init 失败）→ 每个 run 之间必须按 PID 显式 kill。② dpose（关键运动量指标）= 查询间关节位移 L2，比 pelvis_z 更能区分「动/不动」。

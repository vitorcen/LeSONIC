SONIC-VLA 原理 & 数据集制作关键步骤
SONIC-VLA: Architecture Principle & Dataset Pipeline

TL;DR — VLA(GR00T)只产出一个 64 维 FSQ 动作 token(“做什么”),冻结的 SONIC WBC 把 token 解码成 29 个关节目标并兜住平衡(“怎么做”)。机器人是被 token 驱动的,不是被参考动作驱动的;命令行里的 .pkl 只是借用 SONIC 跟踪环境时的出生姿态 + 必填占位,不进驱动回路。数据集制作的核心 = 让冻结 WBC 真去跟踪一段参考、把它每帧吐的 token 录下来,再教 GR00T 从 (画面, 指令) 复现这些 token。
The VLA emits only a 64-d FSQ token (“what”); the frozen WBC decodes it to joint targets and keeps balance (“how”). The robot is driven by the token, not by the reference motion — the .pkl is just the spawn pose + a required placeholder of SONIC’s tracking env, never in the driving loop.

1. 架构原理:token 是“什么”,WBC 是“怎么” / token = what, WBC = how

整条链路把一个难问题拆成两半:

  • 语义/意图层(会变、要泛化)→ 交给 VLA:(自我视角图像 + 文字指令) → GR00T N1.7 → 64 维 FSQ token。token 是 SONIC 动作潜空间里的一个离散码,代表“此刻该摆出的全身动作”。
  • 物理/平衡层(不变、最难、已练好)→ 交给冻结的 GEAR-SONIC WBC:decode(token, 本体感觉) → 29 个关节目标 @50Hz,内部隐式维持站立平衡、脚地接触。

为什么要这么拆?平衡是机器人学里最难、最不该重学的部分。SONIC WBC 是用 4096 个并行环境、海量迭代练出来的通用动作跟踪器。VLA 只需在它的 token 空间里“点菜”,不必从零学会怎么不摔——这就是“架构侧”路线的核心红利。

2. VLA 学会动作了,为什么命令行还要带 .pkl? / why still pass a reference pkl?

这是最容易误解的一点。答案:.pkl 完全没参与驱动机器人。 机器人动起来 100% 靠 GR00T 吐的 token → WBC 解码,参考动作一帧都没喂进解码公式。

那为什么还必须带?因为我们的 in-process demo 借用了 SONIC 的“跟踪(tracking)eval 环境”,而那个环境在架构上为“跟踪一段参考”而生,不加载参考就建不起来。.pkl 只做三件脚手架的事:

The pkl is a stage start-position mark: once the performers (GR00T+WBC) step on it, the act is all theirs.

3. 数据集制作关键步骤 / the dataset pipeline

核心思路:token 不是凭空造的,是“真让冻结 WBC 去跟踪一段参考、把它每帧实际吐的 token 抄下来”。所以制作链路本质是“一次受控的 WBC 跟踪 + 偷取 token”。

① 切窗口 / cut a window

从整段长 clip(fight/run/dance,MimicKit 格式)按秒手挑一段干净动作,转成 SONIC 的 robot_filtered 格式(root_trans/pose_aa/dof/root_rot/fps)。脚本 cut_motion_window.py。手挑而非自动切——自动速度谷切割会切在动作中间或跨转身,标注全错。

② 两道筛查闸门(关键!)/ two screening gates

详见 §4。一句话:先问“严格跟踪能不能全程不被安全阈值打断”,再问“到底有没有真摔”。只有“没真摔”的窗口才值得录——否则 token 编码的是 WBC 跟不住的姿态,VLA 学了会出不稳行为。

③ 录 token(偷取的核心)/ record tokens

让冻结 WBC 物理跟踪这段参考,同时挂钩子(hook)截下它每帧吐的 token:

  • 钩在 actor_module.decode("g1_dyn", d),取 d["token_flattened"] → 64 维,正是 C++ 部署里发布的 token_state。
  • 同时录 ego 相机帧 + 本体感觉,存一份 .npz/帧。
  • 录制必须 terms=null:这些窗口会触发 strict 安全阈值(见 §4),不关掉的话 episode 会被中途 reset,token 录一半就断。关掉只影响“是否 reset”,不改 WBC 的动作,所以 token 和部署一致。

④ 转 LeRobot 数据集 / convert

把裸 npz 转成 GR00T 能吃的 UNITREE_G1_SONIC LeRobot v2.1 数据集:action.motion_token(64)、observation.images.ego_view、annotation.human.task_description(=指令)。脚本 sonic_vla_npz_to_lerobot.py。

⑤ finetune GR00T / train

冻结 VLM 主干,只训 DiT 动作头+projector,让模型学会 (ego 图像, 指令) → token。单 4090 ~55 min / 6000 步。判据 = 开环 MSE:落进已知好区间(≤0.004)= 模型确实学会吐对应 token,不是背均值。

⑥ dump 预测 token → flow3 / deploy

从训好的 checkpoint 跑开环、把每个窗口的预测 token 导出,再注入 WBC 在 Isaac 里看效果。两种播放见 §6。

4. 两道筛查闸门:本路线的关键洞察 / the two gates — the key insight

核心洞察 — 严格闸门把“碰了安全边界”当成了“物理不可行”,这俩不是一回事。10 个手挑窗口里严格闸门只过 2 个,但用“到底有没有真摔”重判后6 个从头到尾没摔。
The strict gate uses “tripped a safety boundary” as a proxy for “physically infeasible” — they are not the same. 2/10 pass strict, but 6/10 never actually fall.
① 严格跟踪闸门 / strict② 物理可行性闸门 / phys-valid
问什么冻结 WBC 能不能在 SONIC 自带的严格安全包络内全程跟住?WBC 跟踪时机器人到底有没有真摔?
判据4 个 adaptive 终止项:根高>15cm、躯干倾>25.6°、手脚高>15cm(单帧)、脚踝>20cm(单帧)任一触发即终止关掉那 4 项,只看:根高<0.6m 或 躯干倾>40° = 真摔
脚本gear_sonic_screen.shgear_sonic_phys_screen.sh + phys_valid_screen.py
结果(10 窗口)2/10 过6/10 没摔 + 2 长前缀

为什么差这么多?因为快拳/快踢里,脚踝某一帧超参考 20cm 就触发 foot_pos_xyz 终止——但整体姿态跟得很好(局部姿态误差 mpjpe_l 全部 33–46mm,在分布内),机器人根本没摔。严格闸门是发布级的保守安全监视器,不是“物理可行性判决”。

诚实边界 — “没摔” ≠ “像那个动作”。关掉终止后 WBC 可能漂着站住(如某段 moonwalk 全局漂 ~483mm,可能只是站着晃)。所以:录制判据 = 物理有效(没摔→token 稳定可实现);保真度 = GUI 视觉筛(MSE 好 ≠ 闭环视觉好)。三方评审(codex gpt-5.5 + mimo + 本机)一致认可这套两层判据。

5. token 录制的物理意义 / what a recorded token actually is

  • FSQ 是固定网格:量化器无可学参数,token 值落在固定的 1/16 离散网格上。所以微调 WBC 不会改 token 值本身(但会改 decoder 把 token 解成什么动作)。
  • token 来自 decode 的输入:我们截的是 decode 入参里的 token_flattened,正是部署侧 proprio 里的 token_state。所以录到的就是真 token,不是近似。
  • 终止 ≠ token 坏:只要 WBC 跟踪那一帧局部姿态在分布内(mpjpe_l 低),那帧 token 就是有效训练目标。终止标志是元数据,不污染 token。前缀窗口只录到摔之前那段。

6. flow3 两种播放:离线 loop vs live 实时 / offline loop vs live

🔁 离线 loop🛰️ live(带 server)
怎么动回放预先 dump 的 token,保证流畅GR00T server 真正实时推理在环
GUI不卡会卡顿(ZMQ 每 40 步阻塞主线程)
动作持续性全程忠实短 bootstrap 进动作后,一次性窗口可能渐站住(单帧 obs 无记忆)
脚本gear_sonic_flow3.shgear_sonic_live_demo.sh @flow3
用途看动作效果、录正式视频验证 GR00T 真在环、看实时行为
关于“慢 ~2x” — live/inject 在 GUI 里比运动学回放慢近一倍,不是 bug:token 每步推进 1 个 @50Hz 时序正确,慢的是 live viewer 实时率——inject 每步要解全物理 + 跑 WBC,replay 只写姿态(无物理无策略),Isaac GUI 不锁实时 → inject 每秒画的帧少 → 看着像慢动作,抹平了踢/跑的力量感。要真实速度看保真度,走 render-to-mp4 按 50fps 渲染。
